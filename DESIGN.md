# director AI 影视 Studio UI 约束

## 1. 适配结论

### 结论

`SpaceX` 的原始设计规范不适合直接用于当前项目。

原因很直接：

- 它是品牌展示站语言，不是高频操作的创作工作台语言。
- 它强调全屏摄影、极少交互、零卡片零表单，但当前产品核心正好相反：需要承载项目配置、阶段切换、资产管理、批量操作、异步任务反馈和长文本编辑。
- 当前代码已经形成了以 `glass-*` token、面板、卡片、抽屉、模态框为核心的界面结构，强行改成“文字直接压在大图上”会破坏信息层级和操作效率。

### 可以保留的灵感

- 黑色系、克制、偏“任务台”的氛围。
- 工业感标题和阶段标签。
- 强对比媒体画面作为氛围或内容主角，而不是装饰图片。
- 少而准的强调色，不做花哨渐变堆叠。

### 明确不采纳的部分

- 不做全站 `100vh` 电影式分屏。
- 不做“无卡片、无容器、无表单边界”。
- 不做全量大写字母排版。
- 不做“只有一种 ghost button”。
- 不把真实业务内容放到大图背景上牺牲可读性。

## 2. 产品语境

本项目不是官网，而是 AI 影视创作工作台。UI 的首要目标不是“震撼感”，而是：

1. 让用户快速理解当前项目、当前剧集、当前阶段。
2. 让用户在资产、脚本、分镜、视频、配音之间持续切换时不迷失。
3. 让媒体预览、生成状态、失败原因、可编辑字段同时清楚可见。
4. 让“风格资产”成为一个可浏览、可选择、可复用、可追踪来源的资产类型，而不是一次性输入框。

因此，本规范采用的方向是：

`SpaceX` 气质 -> 收敛为“黑色任务台 + 媒体驱动 + 分层面板 + 高密度但不拥挤”的工作台设计语言。

## 3. 核心设计原则

### 3.1 媒体是内容，不是背景噱头

- 角色图、场景图、风格预览图、分镜图、视频封面可以承担视觉冲击。
- 这些画面应该出现在卡片、预览区、详情头图、空状态插图里。
- 除营销页或少数空状态外，不使用大面积全屏摄影做页面背景。

### 3.2 面板必须存在

- 工作台页面必须通过面板、卡片、抽屉、模态把信息分区。
- 用户编辑、筛选、确认、对比时必须有清晰边界。
- “界面消失”不是目标；“让用户一眼知道哪里能看、哪里能改、哪里在跑任务”才是目标。

### 3.3 层级优先于风格

- 先解决信息层级，再谈视觉气质。
- 标题、阶段、状态、操作、帮助说明必须可快速扫读。
- 任何视觉方案只要降低可读性或误导交互优先级，就视为不合格。

### 3.4 风格资产要像正式资产，不像提示词附件

- 风格卡片必须能区分名称、来源、适用说明、预览图、标签、正向提示词摘要。
- “系统风格”和“用户风格”必须有稳定的视觉区分，但不能喧宾夺主。
- 风格选择控件应优先展示“已选风格资产”，其次才是兼容旧字段的文本风格。

## 4. 视觉方向

### 4.1 总体气质

- 基调：冷静、专业、偏电影制作后台，而不是消费级娱乐产品。
- 观感：深色任务环境里承载高质量媒体内容，像剪辑室或数字制片台。
- 情绪关键词：沉浸、可靠、精密、可控。

### 4.2 页面结构

- 页面背景可以是深色纯底、弱渐变、轻微噪点或极弱光晕。
- 实际内容必须落在面板体系上，不能直接悬浮在复杂背景上。
- 顶部导航、阶段导航、侧边目录、卡片网格、详情模态是默认骨架。

### 4.3 色彩策略

- 主背景应偏深色中性黑，不用纯黑压死层级。
- 内容面板使用比背景略亮的分层底色。
- 强调色只承担焦点、选中、运行中、可点击状态。
- 危险、警告、成功保留清晰语义色，不能被品牌色吞掉。

## 5. 可落地 Token 约束

### 5.1 与现有实现的关系

当前项目已经在使用：

- `src/styles/ui-tokens-glass.css`
- `src/styles/ui-semantic-glass.css`
- `glass-surface`
- `glass-surface-modal`
- `glass-btn-*`
- `glass-input-base`
- `glass-chip`

因此后续 UI 收敛规则如下：

- 优先复用现有语义 token 和语义 class。
- 不允许在业务组件里频繁手写一次性色值替代 token。
- 如果要整体向“深色任务台”迁移，先改 token 层，再改组件，不允许组件各自偷改。

### 5.2 背景与表面

- 页面底：使用深色中性底，允许极轻微动态氛围，但不能影响阅读。
- 一级表面：承载主要表单、卡片、列表。
- 二级表面：承载 hover、选中、辅助容器、预览外框。
- 模态表面：对比更强，确保长表单和确认操作稳定可读。

### 5.3 边框与阴影

- 阴影只用于分层，不用于制造营销感。
- 面板阴影要弱，边框要稳定，避免过度漂浮。
- 焦点态优先靠描边和 ring，不靠强烈发光。

## 6. 排版规则

### 6.1 字体角色

- 标题字体可以保留少量工业感，体现“制作流程”和“任务控制台”气质。
- 正文、表单、长说明必须回到高可读字体，不追求全站军事化大写。

### 6.2 大小写策略

- 页面标题、阶段标签、状态徽标可适度使用大写或加宽字距。
- 正文、描述、表单标签、帮助信息、长提示词内容禁止强制全大写。
- 中文界面不做伪英文式全大写装饰。

### 6.3 信息密度

- 工作台默认是中高密度，不做过度留白。
- 同一块区域里，标题、说明、字段、操作按钮之间要有稳定间距节奏。
- 长文案说明要可折叠、可截断、可展开，不要把所有文字一次性摊开。

## 7. 组件级约束

### 7.1 顶部导航与阶段导航

- 必须稳定悬浮或固定，保证用户始终知道自己在哪个项目、哪一阶段。
- 阶段导航要更像“制作流程胶囊”而不是官网 tab。
- 当前阶段、可进入阶段、运行中阶段、阻塞阶段需要不同状态表达。

### 7.2 资产卡片

- 卡片是资产中心的主表达，不可取消。
- 预览图占卡片视觉主位，但名称、来源、标签、操作不能被图压住。
- 操作按钮应稳定放在卡片底部或 hover 工具条中，避免每张卡布局漂移。

### 7.3 风格资产卡片

- 至少展示：预览图、风格名、来源、简短说明、提示词摘要、标签。
- 只读系统风格与可编辑用户风格要有明确但克制的区别。
- “查看”和“编辑”是主操作，“删除”永远降低一级。
- 提示词摘要最多展示 2 到 3 行，超出走详情查看。

### 7.4 表单与配置区

- 每组配置都必须有标题、用途说明、输入区。
- 高风险配置或会影响下游阶段的配置，要有轻量提示。
- 风格选择和模型选择不应混成一组，因为它们分别代表“视觉约束”和“能力来源”。

### 7.5 异步任务反馈

- 运行中、排队中、失败、成功必须有统一视觉语言。
- 任务状态不能只靠 toast，页面内要有持续可见的状态落点。
- 失败反馈必须优先展示可执行下一步，而不是只显示报错文本。

### 7.6 模态与抽屉

- 创建、编辑、预览、确认删除这些操作继续使用模态或抽屉。
- 模态标题区要明确当前对象和当前动作。
- 长表单模态必须固定底部主操作区，避免滚到底部才看见保存按钮。

## 8. 图片与视频使用规范

### 8.1 允许的用法

- 卡片预览图
- 详情头图
- 风格对比区
- 生成结果预览区
- 空状态或引导区的小型氛围视觉

### 8.2 禁止的用法

- 把整页真实业务内容直接压在复杂图片上
- 为了“电影感”牺牲按钮可见性
- 用大图替代结构化信息
- 在资产列表页引入大量与资产无关的装饰摄影

## 9. 动效原则

- 动效服务于状态切换、面板展开、任务反馈。
- 页面进入、阶段切换、卡片 hover 可以有轻量动效，但必须短、稳、可预测。
- 不做长时间漂浮、过强 blur 变化、无意义视差。
- 运行中状态优先用脉冲、进度、流动边框，不用夸张粒子效果。

## 10. 响应式约束

- 桌面端优先，保证工作台效率。
- 平板端保持主要流程可用，允许信息折叠。
- 移动端不追求桌面完整复刻，优先保留查看、选择、确认、轻编辑。
- 阶段导航、资产筛选、卡片操作在小屏上必须有明确折叠策略，不能简单缩小字号硬塞。

## 11. Do / Don't

### Do

- 用深色任务台承载媒体内容和生产流程。
- 用卡片、面板、模态明确分区。
- 让风格资产成为一个完整对象，而不是一段提示词。
- 把高质量预览图放在用户真正需要判断的地方。
- 保持阶段、状态、操作的稳定层级。
- 尽量通过现有语义 token 和 class 实现新界面。

### Don't

- 不要把展示站语言直接搬进工作台。
- 不要取消卡片和面板边界。
- 不要为了酷炫让文字悬浮在复杂图片上。
- 不要把所有标题都做成全大写英文风。
- 不要在业务组件里到处写新的魔法色值。
- 不要让删除、危险操作和主操作视觉权重相同。

## 12. 给设计和实现代理的简化指令

如果后续要基于这份文档继续出 UI，请默认遵守以下约束：

1. 这是“AI 影视创作工作台”，不是品牌官网。
2. 保留冷静、工业、电影制作后台的气质，不照搬 `SpaceX` 官网布局。
3. 页面必须有面板、卡片、模态、导航层级。
4. 媒体内容出现在资产卡、预览区、详情区，不做全屏背景主叙事。
5. 风格资产是正式资产类型，要突出名称、来源、预览、标签、提示词摘要和复用关系。
6. 复用现有 `glass-*` token 和组件语义；若要切换深色，优先改 token，不要逐页硬写。
7. 正文和表单优先可读性，禁止全站大写排版。
8. 动效只服务于状态和流程，不做营销式表演动画。

## 13. 当前落地建议

这份规范对应当前项目，最合适的执行顺序是：

1. 先把新增的风格资产相关 UI 保持在现有 `glass` 体系内，确保结构正确。
2. 再逐步统一“风格卡片、风格选择器、项目配置里的风格入口、生成阶段中的风格显示”。
3. 如果后续确实要往更深色的任务台方向演进，单独做一次 token 层主题升级，不和业务功能改动混在一起。

这样改动最小，也最不容易把当前工作台做成“看起来像官网，实际不好用”的状态。
